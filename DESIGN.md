# 設計メモ(nantotonomemo)

本システムの設計思想と主要な決定・背景を残すメモ。
仕様の細部は各プロンプト/ファイルに、"なぜそうしたか" はここに書く。

## 上位ゴール
- ユーザーの「なんちゃってデジタルツイン」を作る。
- log.md(履歴) と profile.md(人物像) を蓄積し、好み・価値観・人格を少しずつ推定する。
- 直近の目的: 人格・好みの抽出。

## 全体アーキテクチャ: 2層(採取 / 統合)
- **日次(採取層) = prompt_cron.md / 毎朝3時 JST**
  - 前回 today.html の結果(done/反応/コメント)を log.md に記録し、新しい5提案で today.html を上書き。
  - 保守的に振る舞う: 1コメントで人格を断定しない。反応は参考情報にとどめる。
- **週次(統合層) = prompt_weekly.md / 木曜 01:10 JST (`10 1 * * 4`, 使用量リセット直後)**
  - log.md 全体を読み、profile.md を根拠つきで更新。再現しているシグナルの確信度を上げる。
  - today.html には一切触らない(役割を混ぜない)。
- 分ける理由: 「毎日ちょっとずつ集める」と「週1でまとめて意味づけする」を分離することで、
  過度な一般化を避けつつ、確信度を段階的に育てられる。

## ファイルの役割
- today.html: ユーザーが見る"今日の提案"。1枚だけを in-place で上書き。操作(check/反応/コメント)はここに autosave。
- today_template.html: 固定デザイン。差し替えは日付/status/5タスク/textareaのみ。CSS・レイアウト・スクリプトは変更しない。
- log.md: 過去の結果の蓄積(1日1行の構造化形式)。
- profile.md: 安定的な人物像。根拠つきで保持。
- prompt.md: 元の仕様(日本語)。prompt_cron.md がその運用版。

## 重要な設計判断と背景

### 1. today.html と log.md の役割分担
- today.html = "今の提案"、log.md = "過去の結果"。
- 初期の不具合: cron指示が「提案を log.md に書く」になっていて today.html が9日間更新されず、
  実データも記録されなかった。役割の取り違えが原因。→ prompt_cron.md で明確化。

### 2. 「前回」参照 + 冪等ガード
- 間が空いた場合、"前日"ではなく"リポジトリにある最後の today.html(=前回生成分)"を見る。
- 記録の日付見出しは、その today.html の `<h1>` の日付を使う(実行日ではない)。
- ガード: today.html が既に本日の日付なら再生成しない(同日二重実行で進行中の1日を壊さない)。
- ガード: 同一日付のエントリが log.md にあれば二重追記しない。

### 3. 根拠つき人格(evidence-linked profile)
- profile.md の各記述に 確信度(低/中/高) と 根拠(logの日付) を付ける。
- 診断由来(オンボーディング) と log観測 を区別する。
- 1件だけで断定しない。再現(複数日・複数の言い方)して初めて確信度を上げる。
- 背景: 一度、根拠のない架空データ(存在しない提案日)を作ってしまった失敗があり、
  その再発防止として "証拠(logの日付)に紐づかない記述は書かない" を明文化した。

### 4. 反応(😊😐😕)の扱い
- 😕 でジャンル/形式を避けたり頻度を下げたりしない。「その回のお題が合わなかった」程度に扱う。
- 反応は参考情報。強く効かせるのは、明示的に profile へ昇格した要望だけ。
- 1回のコメントでジャンルやテーマを丸ごと除外しない。繰り返し出て初めて強いルールに格上げ。

### 5. シグナルの定義
- シグナル = 人格推定の手がかり。log.md に溜まる、人となりを示す情報。
  - 行動シグナル: done / 反応(😊😐😕)。ノイズ多め(未実施=嫌い とは限らない)。
  - 言語シグナル: コメント欄。最も濃い(価値観・口調・文脈まで拾える)。
- 採取の工夫(将来): 催促しない設計を保ちつつ、5枠のうち時々1つを"記録系の軽い内省タスク"にして
  自然にコメントを引き出す。

## 運用パラメータ
- 日次: 毎朝3時 JST。
- 週次: 木曜 01:10 JST(`10 1 * * 4`)。使用量リセット直後で枠がフルな時間帯。
- モデル: タスク複雑度で振り分け。定型の日次/週次は Sonnet 5、複雑タスク(設計・多ファイル)は Opus。
  迷ったら Sonnet 5 で回し、品質が足りない工程だけ Opus に上げる。
- プロンプト言語: 英語(トークン節約)。ただし生成物(today.html/log.md/profile.md)は必ず日本語(hard rule)。
- 設計メモ/README など"読み物"は日本語。

## push / commit 運用
- push は git proxy 優先。403 のときは GH_TOKEN で HTTPS フォールバック(prompt_cron.md に記載)。
  トークンはファイル・コミット・出力に絶対残さない。
- commit identity: 自動ルーティンの更新はユーザー名義(tamorian7@gmail.com)、
  Claude の対話セッションでの変更は Claude(noreply@anthropic.com)。
